# DentistCabinet

Веб-приложение для стоматологической клиники: пациенты и карточки, визиты и услуги, сводки по оплатам, правила уведомлений (Telegram через worker), дашборд.

## Стек

- **Frontend:** Next.js 15 (App Router), React 19, TypeScript, Tailwind CSS, Radix/shadcn-паттерн
- **Backend:** Route Handlers (`/api/`*), Node.js
- **БД:** PostgreSQL, Prisma ORM
- **Auth:** email/password, bcrypt, JWT (httpOnly cookie)
- **Фоновые задачи:** `worker/` (обработка `DomainEvent`, Telegram long polling)
- **Деплой:** Docker, docker-compose

## Системные требования

### Защита персональных данных и соответствие ФЗ-152

Приложение обрабатывает персональные данные пациентов (ФИО, контакты, сведения о здоровье). Эксплуатант (клиника / администратор хостинга) обязан обеспечить организационные и технические меры; ниже — что реализовано в продукте и что нужно настроить на стороне инфраструктуры.

**В приложении:**

- Аутентификация по паролю (хеширование **bcrypt**), сессия через **JWT** в **httpOnly**-cookie.
- Доступ к данным **по клинике** (`clinicId`): API проверяет принадлежность сущностей текущему пользователю клиники.
- Журнал **AuditLog** для ряда операций (создание/изменение/удаление, аудит правил и т.д.) — основа для учёта действий с данными.

**Обязательно настроить у эксплуатанта (надёжные копии и шифрование в смысле 152-ФЗ обеспечиваются в том числе инфраструктурой):**

1. **Конфиденциальность при передаче:** публикация сервиса только по **HTTPS** (TLS) в продакшене; для PostgreSQL в облаке — соединение с **SSL** (например, `?sslmode=require` в `DATABASE_URL`, см. `.env.example`).
2. **Шифрование:** шифрование дисков и томов СУБД на стороне хостинга/ВМ; при выгрузке бэкапов на внешние носители — шифрование архивов или защищённое облако с шифрованием.
3. **Резервное копирование:** регулярные снимки базы (**pg_dump** или снапшоты провайдера), хранение отдельно от продакшена, **периодическая проверка восстановления** на тестовой среде. Вспомогательный скрипт:
  ```bash
   chmod +x scripts/backup-db.sh
   DATABASE_URL="..." ./scripts/backup-db.sh
  ```
   Каталог выгрузки — переменная `BACKUP_DIR` (по умолчанию `./backups`). Нужен установленный клиент PostgreSQL (`pg_dump`).
4. **Управление доступом:** учётные записи только для уполномоченных сотрудников; при увольнении — отзыв доступа; не коммитить `.env` с секретами.
5. **Локальные акты клиники** (политика ПДн, согласия) — по требованиям юридического сопровождения, вне репозитория.

Код не заменяет юридическую экспертизу; при работе с медицинскими данными учитывайте отраслевые нормы.

## Быстрый старт (локально)

1. Скопируйте переменные окружения:
  ```bash
   cp .env.example .env
  ```
   Задайте `JWT_SECRET` (не короче 32 символов) и при необходимости `DATABASE_URL`.
2. Поднимите PostgreSQL и примените миграции:
  ```bash
   npx prisma migrate deploy
  ```
3. Установите зависимости и запустите dev:
  ```bash
   npm install
   npm run dev
  ```
4. Откройте [http://localhost:3000](http://localhost:3000), зарегистрируйте клинику или используйте сид:
  ```bash
   npm run db:seed
  ```
   Демо-аккаунт после сида: `demo@clinic.local` / `demo-demo`

### Доступ с телефона или другого ПК в той же Wi‑Fi сети

По умолчанию `next dev` слушает только `localhost`, поэтому с другого устройства сайт не открыть. Чтобы открыть сервис по локальной сети:

1. Узнайте **LAN‑IP** компьютера, где запущен проект (например `192.168.1.42`): в Linux `ip a` или настройки Wi‑Fi в системе.
2. Запустите dev‑сервер на всех интерфейсах:
  ```bash
   npm run dev:lan
  ```
   (это то же самое, что `next dev -H 0.0.0.0`.)
3. На телефоне или втором ПК в браузере откройте `http://<LAN-IP>:3000` (порт как у Next.js, обычно 3000).
4. Если порт недоступен, разрешите входящие подключения в **файрволе** ОС для Node/next (на Ubuntu для теста: `sudo ufw allow 3000/tcp`).

Важно: трафик идёт по **HTTP без шифрования** внутри вашей сети — для реальных персональных данных в продакшене нужен обычный сервер с **HTTPS** и защищённая сеть. Для локальной провёрстки интерфейса на телефоне этого достаточно.

При ссылках из Telegram и т.п. может понадобиться выставить в `.env` `**APP_URL`** на тот же `http://<LAN-IP>:3000`, иначе бот может подставлять неверный базовый URL.

Для собранного приложения (`npm run build` и `npm start`) по умолчанию тоже только localhost; для доступа из сети: `npx next start -H 0.0.0.0`.

### Сводки по деньгам

- Таблица **по дням** (`/api/reports/daily`): суммы по **строкам оплаты визита** (`VisitPaymentLine`); день строки определяется по **дате приёма** визита (`visit.startAt`), с фильтром по статусам из настроек.
- Колонки: **Всего**, **Терминал** (безнал через терминал/карту — способы из `terminalMethods`, **кроме** банковских переводов), **Переводы** (только `TRANSFER`), **Наличные** (`cashMethods`, по умолчанию `CASH`). Способ `OTHER` учитывается в «Всего», если не попал в указанные группы.
- Доли **сплита** (например 65% и 35%) задаются на странице «Сводки» и сохраняются в `Clinic.settings.moneyReport` (через `PATCH /api/clinic/settings`). Списки `terminalMethods` и `cashMethods` можно переопределить в JSON настроек клиники.

### Правила уведомлений

1. Убедитесь, что запущен **worker** (`npm run worker` или сервис `worker` в Docker) и заданы `TELEGRAM_BOT_TOKEN`, при необходимости `APP_URL`.
2. В **Настройках** получите ссылку привязки Telegram и откройте её в мессенджере.
3. В **Правилах** создайте правило:
  - **Событие-триггер** — когда проверять правило (создание визита, платёж и т.д.).
  - **Текст** — шаблон с подстановками `{{patient_name}}`, `{{visit_date}}`, `{{clinic_name}}` и др.
  - **Дерево условий** — JSON по модели из `src/lib/rule-engine.ts`:
    - Узлы: `and` / `or` / `not` / `cmp`.
    - Для `cmp`: поля `left` / `right` как `{ "kind": "field", "path": "patient.status" }` или `{ "kind": "literal", "value": ... }`, операции `eq`, `neq`, `gt`, `contains`, `is_empty`, …
  - Пример «всегда срабатывать» (для теста):
    ```json
    {"op":"cmp","left":{"kind":"literal","value":1},"cmp":"eq","right":{"kind":"literal","value":1}}
    ```
4. Worker подставляет контекст (`patient`, `visit`, `clinic`) и отправляет сообщение в Telegram, если условие выполнено.

Отладка условия: `POST /api/rules/[id]/evaluate` с телом `{ "context": { "patient": { "status": "ACTIVE" } } }`.

### Услуги в визите

- В справочнике **Услуги** задаются цена по умолчанию и себестоимость.
- В карточке **визита** можно добавить любую услугу в любой момент (в том числе после завершения визита), изменить **цену и количество** для этой строки, удалить строку. Итоги визита пересчитываются автоматически (`recalcVisitTotals`).

### Регистрация и AuditLog

Запись в `AuditLog` при создании клиники выполняется **после** успешной транзакции Prisma, чтобы `clinicId` уже существовал во внешнем ключе.

## Docker Compose

```bash
cp .env.example .env
docker compose up --build
```

Сервисы: `db`, `web`, `worker`. Каталог `prisma/` в образе копируется до `npm ci` (см. `Dockerfile`).

**Один образ на `web` и `worker`:** собирается только сервис `web` (тег `dentist-cabinet:local`), `worker` подключает тот же образ. Раньше два параллельных `npm ci` перегружали сеть и выглядели как «бесконечная» сборка.

**BuildKit:** в `Dockerfile` используется кэш npm (`RUN --mount=type=cache,...`). Нужен Docker с BuildKit (по умолчанию в актуальных версиях). При ошибке синтаксиса: `DOCKER_BUILDKIT=1 docker compose build`.

### Сборка: сеть и registry

При `ETIMEDOUT` к registry — повторить сборку; на Linux при сетевых сбоях можно попробовать `docker compose -f docker-compose.yml -f docker-compose.build-host.yml build` (сеть хоста только для шага сборки `web`). Не задавайте `NPM_REGISTRY` на несовместимое зеркало без необходимости.
